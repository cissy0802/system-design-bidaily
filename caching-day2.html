Day 2 Medium Caching LRU/LFU/ARC Invalidation

缓存 — 从浏览器到 DB 的多层穿透艺术Caching: Layers, Eviction Policies, Write Strategies, Invalidation

问题场景

用户访问首页 200ms 内必须返回，DB 查询一次需要 80ms，单查首页要 7 个查询——不缓存就是死。但缓存不是『加个 Redis』那么简单：Facebook 2010 年因为一次 Memcached 失效风暴丢了 2.5 小时全站可用性；Discord 2017 年发现消息列表的 LRU 在热点频道下表现糟糕，换 LFU 后命中率从 78% 提到 96%；GitHub 2020 年 PR 页面用 Russian Doll caching，命中率 99.3%。

本期讲三件事：缓存放在哪一层、怎么淘汰、怎么写。最关键的是『缓存失效』，被 Phil Karlton 称为计算机科学两大难题之一（另一个是命名）。

需求与约束（面试必问）

  • 读写比：100:1 适合 cache，1:1 写多读少基本没收益（缓存命中率太低）。
  • 数据热度分布：是否 80/20 Pareto？长尾分布缓存效果差，热点严重则单 key 风暴。
  • 容忍 stale 多久：用户头像 1 小时旧没事；账户余额 1 秒旧也是事故。
  • 对象大小：1KB 还是 1MB？Memcached 默认 slab 上限 1MB，超过要切片或换 Redis。
  • 命中率目标：< 80% 通常意味着缓存层没价值；> 95% 才能真正卸载后端。
  • 失效策略：TTL、显式 invalidate、还是 write-through？

关键技术点

1. 缓存层次：Browser / CDN / App / DB Buffer Pool

原理：每一层缓存解决不同的延迟问题。Browser cache 省网络（0ms vs 50ms）；CDN 把静态资产放到用户 50 公里以内的 PoP，p50 延迟 10ms；app 进程内 cache（Guava / Caffeine / Python functools.lru_cache）省 Redis 一次 RTT（0.1ms vs 1ms）；Redis 跨进程共享，命中后省 DB 查询（1ms vs 50ms）；DB buffer pool（Postgres shared_buffers, InnoDB buffer pool）省磁盘 IO。

Trade-off：
  • 进程内 cache：✅ 0.1ms 级延迟、无网络故障；❌ N 个进程 N 份副本（内存浪费）、失效要广播（很难做对）。
  • 分布式 cache (Redis)：✅ 一致视图、好失效；❌ 多一次 RTT、Redis 本身要扩容/HA。
  • CDN：✅ 全球分布、抗 DDoS；❌ 只能缓存可公共复用的内容，失效慢（Cloudflare 全球 purge ~30s）。
现实案例：
  • Facebook：TAO（in-memory graph cache）+ Memcached 海洋（数万台），命中率 > 99%，每秒处理 10 亿+ 次读。论文《Scaling Memcache at Facebook》是必读。
  • Twitter：用户的 timeline 整页用 Redis cache，热门用户被 fanout 预生成。
  • GitHub Rails：用 Russian Doll caching（嵌套片段缓存），PR 详情页命中率 99.3%。
  • Stack Overflow：单数据中心 9 台服务器扛全球访问，靠 5 层缓存（HTTP / Redis / SQL Server buffer / .NET CLR / 客户端）。

2. 淘汰策略：LRU vs LFU vs ARC vs TinyLFU

原理：缓存容量有限，必须选一个『谁先死』规则。

策略淘汰对象适合场景命中率
LRU最久未访问的时间局部性强（最近访问 = 即将再访问）中
LFU访问次数最少的频率局部性强、热点稳定高（稳态）
ARC自适应 LRU+LFU访问模式变化、不想调参高
W-TinyLFU带准入控制的 LFU有突发流量、扫描型访问极高
FIFO/Random先进/随机O(1) 极简、纯 streaming低
Trade-off：
  • LRU：✅ 实现简单（哈希表 + 双向链表 = O(1)）；❌ 被一次性扫描污染（『扫描抗性差』），如有人爬全站。
  • LFU：✅ 抗扫描；❌ 老数据『沉淀』下来，新热点进不来（需要 aging / 衰减计数）。
  • ARC：✅ 自适应；❌ IBM 专利（Linux 因此用 2Q 替代）、实现复杂。
  • W-TinyLFU（Caffeine 用）：✅ 用 Count-Min Sketch 估频率，准入控制让稀有 key 进不来污染；❌ 概率结构有误差。
# Python LRU 极简实现 (用 OrderedDict)
from collections import OrderedDict
class LRU:
    def __init__(self, cap): self.cap, self.d = cap, OrderedDict()
    def get(self, k):
        if k not in self.d: return None
        self.d.move_to_end(k)        # 最近访问 → 末尾
        return self.d[k]
    def put(self, k, v):
        if k in self.d: self.d.move_to_end(k)
        self.d[k] = v
        if len(self.d) > self.cap:
            self.d.popitem(last=False)  # 淘汰头部（最久未访问）

# Redis 实际用『近似 LRU』：每次淘汰从 5 个随机 key 中挑最久的
# maxmemory-policy allkeys-lru / allkeys-lfu / volatile-ttl
现实案例：
  • Redis：默认 allkeys-lru，但实现是近似 LRU（采样 5-10 个 key 比较，节省维护链表的开销）。Redis 4.0+ 支持 LFU。
  • Caffeine (Java)：用 W-TinyLFU，被 Cassandra、HBase、Kafka Streams 默认采用，命中率比 LRU 高 5-15%。
  • Linux Page Cache：用 2Q（active/inactive 双链表），近似 LRU 但抗扫描污染。
  • Discord：消息缓存早期用 LRU，热门频道下『最近未访问』误判严重，切到 LFU 后命中率 78% → 96%。

3. 写策略：Cache-aside vs Write-through vs Write-back

原理：写数据时缓存和 DB 谁先谁后、谁负责，决定了一致性和复杂度。

Cache-aside (lazy)Write-throughWrite-back
读miss → DB → 回填 cachecache hit（永远）cache hit
写写 DB → 删 cache写 cache → 同步写 DB只写 cache → 异步刷 DB
一致性有窗口（最终一致）强（cache = DB）弱（宕机丢未刷数据）
写延迟低（只写 DB）高（两次写）极低（只写内存）
适用Redis + DB 最常见金融、强一致读CPU cache、SSD FTL
# Cache-aside 标准写法 (Python)
def get_user(uid):
    user = redis.get(f"u:{uid}")
    if user: return json.loads(user)
    user = db.query("SELECT * FROM users WHERE id=%s", uid)
    redis.setex(f"u:{uid}", 3600, json.dumps(user))  # TTL 兜底
    return user

def update_user(uid, data):
    db.update("UPDATE users SET ... WHERE id=%s", uid, data)
    redis.delete(f"u:{uid}")   # ⚠️ 删而不是更新，避免并发写错乱
    # ⚠️ 还要小心: 删之后立刻有人读, 可能再次回填旧值 → 双删 / 延迟双删
Trade-off：
  • Cache-aside：✅ 简单、cache 挂了不影响写；❌ 第一次 miss 慢、需手动 invalidate（容易写错）。
  • Write-through：✅ cache 永远是真相；❌ 每次写都贵；cache 挂了写不进去。
  • Write-back：✅ 最快；❌ cache 宕机 = 丢数据，账户余额绝不能用。但 CPU L1/L2 cache、SSD FTL 都用这个。
现实案例：
  • Instagram：feed 用 cache-aside + Redis，写帖子后删相关 cache key。
  • LinkedIn：消息中心用 write-through 到 Voldemort（自家 KV），保证读总是最新。
  • MySQL InnoDB：buffer pool 是 write-back，靠 redo log + checkpoint 保证持久化。
  • Apple iCloud Photos：缩略图用 write-through 到 S3，保证 CDN 拉取永远有效。

4. 失效难题：TTL、惊群、雪崩、穿透

原理：四个经典坑，每个都在生产环境炸过。

  • Thundering herd（缓存击穿）：一个热 key 过期瞬间，1 万个请求同时穿到 DB。解法：互斥锁（single-flight）、逻辑过期（永不真过期，后台异步刷）、提前续约。
  • Cache avalanche（雪崩）：大批 key 同一时刻过期，DB 突然被打爆。解法：TTL 加随机抖动（3600 + random(0, 300)）。
  • Cache penetration（穿透）：查询不存在的 key，每次都回源。解法：缓存空值（短 TTL）、Bloom Filter 前置判断。
  • Stale-while-revalidate：返回旧值同时后台刷新（CDN / HTTP / Next.js ISR 都用），延迟 0、最终一致。
# Single-flight 防击穿 (Go)
var g singleflight.Group
func GetHot(key string) (any, error) {
    v, err, _ := g.Do(key, func() (any, error) {
        if v := redis.Get(key); v != nil { return v, nil }
        v := db.Query(key)
        redis.SetEX(key, v, ttlWithJitter(3600))
        return v, nil
    })
    return v, err
}
现实案例：
  • Facebook Memcached lease：miss 时只发一个 lease token 给一个客户端去回源，其他人等待。论文中详述。
  • Cloudflare：用 stale-while-revalidate，过期内容继续返回 60s 同时后台拉新，p99 几乎无毛刺。
  • Pinterest：曾因 Memcached 全集群同 TTL 雪崩挂掉 30 分钟，事后加 jitter 修复。
  • Vercel ISR：Next.js Incremental Static Regeneration 就是 stale-while-revalidate 的产品化。

扩展与优化（增长后怎么办）

  • 命中率上不去：分析访问分布。如果是长尾，考虑预热（pre-warming）或更大容量；如果是热点，分片到多 key（user:123:part:{0..15}）。
  • Redis 单实例顶不住：上 Redis Cluster（16384 slot）或 Twemproxy / Codis 中间层；写热点要 consistent hashing。
  • 跨区延迟：每区独立 Redis 集群，DB 写后异步广播 invalidation（Facebook 用 mcsqueal）。
  • 对象太大：换 Redis（支持 512MB / value）或切片；考虑只缓存 ID 列表，详情仍走 DB（节省内存）。
  • 瓶颈识别：监控『命中率』『p99 延迟』『回源 QPS』，命中率下跌往往是新功能没缓存或 invalidation bug。

常见陷阱（资深工程师也常翻车）

1. 更新缓存还是删除缓存？ 答案永远是删除。更新会让两个并发写产生乱序（A 写值 v1 → B 写值 v2 → B 更新 cache → A 更新 cache，cache 永远停在 v1）。
2. 先写 DB 还是先删 cache？ 标准是『先 DB 后删 cache』。先删 cache 的话，删完到 DB 写完之间有窗口，别人读会回填旧值。极端情况用『延迟双删』：删 → 写 DB → sleep 500ms → 再删一次。
3. 缓存 vs 数据库的最终一致窗口：永远不可能 0。承认这一点，给业务定『stale 容忍度』。
4. 把 Redis 当 DB 用：Redis 持久化（RDB / AOF）不是设计目标，宕机 RPO 至少几秒。账户余额这种数据，Redis 是 cache 不是 source of truth。
5. 忽略冷启动：服务重启或 cache cluster 替换瞬间，命中率 0%，DB 直接被打爆。要预热或慢启动放量。

English Summary

Caching is a multi-layer art spanning browser, CDN, reverse proxy, in-process, distributed (Redis/Memcached), and DB buffer pools. Choose eviction policy by access pattern: LRU for recency, LFU for frequency, W-TinyLFU (Caffeine) when in doubt. Write strategies trade consistency for performance: cache-aside is the default, write-through ensures correctness, write-back maximizes throughput at the cost of durability. The hardest part is invalidation — beware thundering herd, avalanche (use TTL jitter), penetration (bloom filter), and the eternal question of delete vs update cache (always delete).
