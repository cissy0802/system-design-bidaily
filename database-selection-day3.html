Day 3 Medium Database SQL/NoSQL ACID/BASE OLTP/OLAP

数据库选型 — 不要把所有数据塞进一个 PostgresDatabase Selection: SQL vs NoSQL, OLTP vs OLAP, ACID vs BASE, and When to Pick What

问题场景

『我们用什么数据库？』是系统设计面试里最容易答错的问题。新手默认 MySQL，老手默认 Postgres，更老的工程师会反问：你的工作负载是什么？Uber 2016 年从 Postgres 切回 MySQL（核心调度），又把行程数据迁到 Schemaless（自研 MySQL 上的 KV 层）；Pinterest 把 200+ 个 MySQL 分片当 KV 用，外加 HBase 做时间线；Discord 2017 年从 MongoDB 迁到 Cassandra（120M 消息/天），2023 年再迁到 ScyllaDB（C++ 重写的 Cassandra 兼容，p99 从 40-125ms 降到 15ms）。

本期讲清四件事：SQL 还是 NoSQL、OLTP 还是 OLAP、ACID 还是 BASE、具体选哪家。最关键的判断不是『哪个数据库最强』，而是『你的访问模式、一致性要求、规模和团队能力是什么』。

需求与约束（面试必问）

  • 访问模式：点查（by key）、范围查询、全文搜索、聚合分析、图遍历？这是第一性问题。
  • 读写比与 QPS：100:1 读重 vs 1:10 写重 vs 1:1 写多读少，选型完全不同。
  • 数据规模：100GB、10TB、还是 PB？10TB 以上单机 Postgres 痛苦，PB 级别 OLAP 该上 lakehouse。
  • 一致性要求：账户余额（强一致）、社交点赞（最终一致几秒可接受）、推荐结果（分钟级）？
  • Schema 灵活性：字段会频繁加吗？多租户每个 tenant 自定义字段吗？JSON / document store 更友好。
  • 事务范围：单行原子（KV 即可）、单分区多行（DynamoDB tx 够用）、跨分区（必须 Postgres / Spanner）？
  • 团队能力：3 个人的小团队不要上 Cassandra；没 DBA 的团队不要自建 sharded MySQL。运营成本和雇佣市场也是技术选型。

关键技术点

1. SQL vs NoSQL：本质是 schema 与 join 的取舍

原理：SQL（关系型）的核心能力是强 schema + 任意 join + 二级索引 + ACID 事务。NoSQL 不是一类，是『反 SQL 选择』的总称，至少分四种：

  • KV store（DynamoDB / Redis / RocksDB）：只支持 by-key 操作，O(1) 查询。
  • Document（MongoDB / Couchbase）：JSON-like 文档，schema 灵活，单文档原子。
  • Wide-column（Cassandra / HBase / ScyllaDB / Bigtable）：『行 × 动态列族』，按 partition key 分布，写优化（LSM tree）。
  • Graph（Neo4j / TigerGraph）：节点-边-属性，多跳关系遍历比 SQL join 快几个数量级。
Trade-off：
  • SQL：✅ 关系建模灵活、ad-hoc 查询、强一致 + 跨行事务；❌ 单机扩展上限（垂直）、schema 变更痛苦（大表 ALTER 锁表）、跨库 join 几乎不可能。
  • NoSQL：✅ 水平扩展、schema 灵活、写性能高（LSM）；❌ 没有 join（需 app 端拼装）、二级索引弱、事务有限、ad-hoc 分析困难。
  • 『Postgres 已经够好』论：在 1-10TB 内 + 单数据中心 + QPS 几万以内，Postgres + JSONB + pg_trgm + pgvector + Citus 基本可以替代多数 NoSQL。Notion、Figma、Stripe 早期都是单 Postgres（或分片 Postgres）扛全公司。
现实案例：
  • Notion：单 Postgres 扩到 480+ 个 shard（每个 shard 是独立 RDS），存储所有 block。访问模式都是 by-id 树查询，不需要复杂 join，但需要事务（block 移动要原子）——所以仍坚持 Postgres。
  • Figma：2024 年公开了从单 Postgres 横向分片的工程历程，分片键是 organization_id（多租户天然边界）。
  • Discord：消息走 ScyllaDB（wide-column，按 channel_id 分区，按时间排序），用户/服务器元数据走 Postgres（强一致需求）。
  • Airbnb：核心订单 MySQL，搜索 Elasticsearch，房源详情 Cassandra，分析 Druid + Presto。

2. OLTP vs OLAP：行存 vs 列存的物理差别

原理：OLTP（在线事务处理）和 OLAP（在线分析处理）是两类访问模式，需要完全不同的存储布局。

OLTPOLAP
典型查询『查用户 123 的最近 10 单』『过去一年各国销售额分组』
数据访问点查 + 小范围，少量行 × 多列全表扫描，大量行 × 少列聚合
存储布局行存（一行紧凑在一起）列存（一列连续，便于压缩+向量化）
写入高频小事务（每秒几千~几万）批量 ETL / streaming 注入
延迟ms 级秒~分钟级
代表Postgres, MySQL, DynamoDBSnowflake, BigQuery, ClickHouse, DuckDB
Trade-off：
  • 行存（OLTP）：✅ 单行读写快、事务友好；❌ 聚合扫描时 IO 浪费（要读整行只为算一列）。
  • 列存（OLAP）：✅ 同列数据连续 → 压缩率 5-10×、SIMD 向量化、聚合极快；❌ 单行 update 慢（要改 N 个列文件）、不适合点查。
  • HTAP（混合）：TiDB、SingleStore、CockroachDB 试图统一，做法是同表两份存储（row + column）后台同步。代价是写放大 + 复杂度。多数公司还是用 OLTP + CDC + OLAP 分开。

现实案例：
  • Shopify：核心订单 MySQL（pod 架构，每个 pod 一套独立 MySQL），分析全部走 BigQuery + dbt。
  • Netflix：用 Iceberg + S3 + Presto 做 PB 级 lakehouse，每天数百万查询。
  • Stripe：核心 ledger 是 Mongo（早期）/ 自研系统，分析全在 Snowflake；财务报告必须用快照保证可重现。
  • ClickHouse at Cloudflare：每秒 6M+ HTTP 日志写入，做实时分析；ClickHouse 是列存 + LSM 的代表。

3. ACID vs BASE：强一致和最终一致的代价

原理：ACID（Atomicity, Consistency, Isolation, Durability）是事务保证四件套；BASE（Basically Available, Soft state, Eventual consistency）是大规模分布式系统对一致性的妥协。

  • Atomicity：事务要么全部成功要么全部失败。
  • Consistency：事务前后数据库约束都满足（不是 CAP 里的 C，是数据库约束意义上的）。
  • Isolation：并发事务互不干扰——但有四个隔离级别（Read Uncommitted / Read Committed / Repeatable Read / Serializable），默认通常是 Read Committed，Postgres 是。
  • Durability：commit 后即使宕机也不丢——靠 WAL（write-ahead log）+ fsync。
Trade-off：
  • ACID 的代价：跨节点事务需要 2PC / Raft / Paxos 共识，延迟从 ms 到几十 ms；跨地域事务延迟到几百 ms（光速决定）。Google Spanner 用 TrueTime + Paxos 做全球 ACID，代价是部署复杂度极高（需要原子钟+ GPS 同步）。
  • BASE 的代价：开发者要自己处理冲突、最终一致窗口期、补偿事务（Saga）。复杂度从基础设施挪到业务代码。
  • 隔离级别陷阱：Read Committed 下会有不可重复读（同事务两次读结果不同）；Repeatable Read 下还可能有幻读。MySQL InnoDB 的 RR 用 next-key lock 防幻读，Postgres 的 RR 是快照隔离（Snapshot Isolation）有写偏（write skew）问题。Serializable 是唯一安全的，但吞吐量低。
-- Postgres 写偏（write skew）经典例子：医院值班调度
-- 约束：必须至少 1 个医生在值班
-- 当前有 2 个医生 Alice、Bob 都 on_call=true
BEGIN ISOLATION LEVEL REPEATABLE READ;
-- Tx1 (Alice 想下班):
SELECT count(*) FROM doctors WHERE on_call=true;  -- 看到 2
UPDATE doctors SET on_call=false WHERE name='Alice';
COMMIT;
-- Tx2 (Bob 同时想下班，并发):
SELECT count(*) FROM doctors WHERE on_call=true;  -- 也看到 2
UPDATE doctors SET on_call=false WHERE name='Bob';
COMMIT;
-- 结果：两人都下班了，约束被破坏 → 必须用 SERIALIZABLE 或 SELECT FOR UPDATE
现实案例：
  • Stripe / Square：支付必须 ACID，金额一分都不能丢；用 Postgres + 跨表事务 + idempotency key。
  • DynamoDB：早期只支持单 item 原子（基本 KV），2018 年加了 TransactWriteItems 支持最多 100 个 item 跨分区事务（贵：消耗 2× 容量）。
  • Cassandra：BASE 模型，写默认 quorum，开发者用 lightweight transaction (LWT) 做 compare-and-set，但 LWT 慢 4-5×。
  • CockroachDB / Spanner：试图『鱼与熊掌』——全球分布 + ACID，用 Raft + HLC（Hybrid Logical Clock）。Spanner 全球事务 commit 延迟约 10ms（同区）到 100ms+（跨洲）。

4. 具体选型：Postgres / DynamoDB / Cassandra / Redis 各自的甜区

原理：不要争『哪个最好』，每个数据库的内部结构决定了它的甜区（sweet spot）和踩雷区。

数据库存储引擎甜区踩雷区
Postgres行存 B-tree + MVCC + WALOLTP、复杂查询、JSONB、扩展生态（PostGIS, pgvector）单机写吞吐瓶颈（~10w TPS）、超大表 ALTER、跨库分布式事务
MySQL InnoDB行存 B+ tree + redo log简单 OLTP、读重负载（replica 多）、互联网经典选择复杂查询能力弱、没有 native UUID、JSON 支持差于 PG
DynamoDBSSD KV + B-tree per partitionmassive scale、可预测延迟（个位数 ms p99）、serverless查询表达力弱（只能按 key/sort key 范围）、GSI 限制多、贵
Cassandra / ScyllaLSM tree（写优化）写重负载（百万级 QPS）、时序数据、宽列模型没有 join、二级索引性能差、运维难（compaction、tombstone）
MongoDBWiredTiger（B-tree / LSM 可选）schema 频繁变、JSON 原生、单文档原子跨文档事务慢、关系建模笨拙、过去口碑不佳（4.0+ 已改善）

Snowflake / BigQuery列存 + 分离存储计算PB 级 OLAP、按查询付费、SQL 兼容不适合 OLTP、写延迟高（不能秒级看到新数据，除非用 streaming）
# 决策树：当我开始一个新功能时
# 1. 是单表 by-key 查询，QPS 极高？ → Redis 或 DynamoDB
# 2. 是关系建模 + 需要事务 + ad-hoc 查询？ → Postgres
# 3. 是写远多于读、按时间序、TB+ 数据量？ → Cassandra / Scylla
# 4. 是全文搜索 / 模糊匹配？ → Elasticsearch（搭配 PG/MySQL 作真相源）
# 5. 是报表/BI/分析？ → Snowflake / BigQuery / ClickHouse
# 6. 是图谱关系（社交、欺诈检测）？ → Neo4j 或 PG + recursive CTE
# 7. 不确定？ → Postgres，可以走很远
现实案例：
  • Uber 调度：曾从 Postgres 迁到 MySQL（Schemaless 层），核心原因是分片复制、连接数和运维成本（详见 Uber engineering blog 2016）。
  • Slack 消息：MySQL 分片（按 team_id），channels 表 + messages 表，跨表事务在单 shard 内完成。
  • Pinterest：MySQL 当 KV 用（id → JSON），HBase 存时间线。早期没用 Cassandra 因为团队不熟 Java 运维。
  • Reddit：早期把 Postgres 当『EAV』表用（entity-attribute-value），后来痛苦迁移；教训：不要拿关系数据库当 KV 用。

扩展与优化（增长后怎么办）

  • 单机 Postgres 顶不住：先垂直扩（更大的实例 + 更快 SSD），其次读 replica（写还在主），最后分片（Citus / Vitess / 自研按 tenant_id 分）。
  • 表太大 ALTER 锁表：用 pt-online-schema-change（MySQL）或 pg-osc（Postgres），后台双写迁移，几小时无锁完成。
  • 分析查询拖垮 OLTP：搭 CDC + 数仓，业务侧绝不在生产主库跑 GROUP BY。
  • 多区域：跨区强一致几乎不可能（光速延迟）。要么接受最终一致（Cassandra multi-DC）；要么用 Spanner / CockroachDB（贵但能做）；要么按区域分片（每个用户只属于一个区域）。
  • 成本：DynamoDB 写 1 RCU/WCU 看似便宜，QPS 上百万会发现账单远超自建 Postgres。但加上 DBA 成本就不一定了。

常见陷阱（资深工程师也常翻车）

1. 『一个数据库走天下』：把 Postgres 当缓存、当搜索引擎、当消息队列。每个职责都不专业，最后被规模逼着拆。可以早期偷懒，但要在架构图上承认这是技术债。
2. 把 Redis 当数据库：Redis 持久化（AOF + everysec）丢的就是最近 1 秒数据；订阅消息丢就丢了。账户余额、订单这类数据，Redis 是 cache 不是真相源。
3. 把 MongoDB 当关系数据库：嵌套文档变成 100 层，跨文档事务勉强能做但慢，最后还是要 join。如果你天然在做关系建模，用 Postgres。
4. NoSQL 二级索引随便建：DynamoDB GSI 每个都是独立的表结构，写放大 N 倍，且最终一致（GSI 是异步同步）。Cassandra 的二级索引在大数据集上几乎不可用，要建反查表（denormalize）。
5. 忽略隔离级别：默认 Read Committed 不防写偏，金融场景必须显式 SET TRANSACTION ISOLATION LEVEL SERIALIZABLE 或 SELECT FOR UPDATE。面试时被问『为什么余额会变负』，多半就是隔离级别没设对。
6. 用 OLTP 跑分析：生产 Postgres 主库被一个产品同学的 SELECT COUNT(*) 干掉，是真实故事不是段子。

English Summary

Database selection is not about picking the "best" — it's about matching workload to engine. SQL (Postgres/MySQL) wins for relational modeling, ad-hoc queries, and ACID transactions; NoSQL splits into KV (DynamoDB/Redis), document (MongoDB), wide-column (Cassandra/Scylla), and graph (Neo4j), each optimizing a specific access pattern. OLTP (row store, point queries, ms latency) and OLAP (column store, aggregations, seconds latency) need physically different layouts — never run analytics on your primary OLTP DB; pipe via CDC to a warehouse. ACID guarantees come at the cost of distributed coordination (2PC/consensus); BASE pushes complexity to application code. When in doubt, start with Postgres — it scales further than most teams realize.

深入思考（点击展开答案）

1. 为什么 Cassandra 写比读快，而 Postgres 读比写快？这跟存储引擎选型怎么联系？

本质：B-tree 是原地更新，LSM tree 是追加写。

  • Postgres / MySQL B-tree：写要找到目标页（可能要从磁盘读入）、修改、再写回。Postgres 因 MVCC 实际是写新版本+老版本标记，但页位置固定。读时一次树遍历（3-4 层）就能定位，IO 少。
  • Cassandra LSM tree：写永远是顺序追加到 memtable（内存），满了 flush 成 SSTable（只读不可变）。一次写 = 一次内存操作 + 顺序磁盘写，所以极快（百万 TPS 单机可达）。读时要查 memtable + 可能多个 SSTable，要 bloom filter 加速但仍需多次 IO，所以读慢。

二阶代价：

  • LSM 需要后台 compaction（合并 SSTable）来限制 SSTable 数量；compaction 吃 IO 和 CPU，调不好会让 p99 抖动。Cassandra 运维痛点之一。
  • LSM 的 写放大（write amplification）：一条数据可能被 compaction 重写 5-10 次；B-tree 的写放大主要来自 WAL 双写，相对稳定。
  • LSM 的 读放大（read amplification）：一次读要查 N 个 SSTable；level compaction 把读放大限制在 ~log N。

选型暗示：写远多于读 → LSM（Cassandra/Scylla/RocksDB/HBase）；读多写少 + 复杂查询 → B-tree（Postgres/MySQL）。所以 Discord 消息选 Cassandra（写多读少，每条消息只被读 ~1 次），用户档案选 Postgres。

2. 一个支付系统：用户 A 转 100 元给 B。如果只有 DynamoDB（无跨分区事务）你怎么实现？如果允许用 Postgres 呢？两种方式的故障窗口和补偿成本？

方案 A: Postgres 单库事务：

BEGIN;
UPDATE accounts SET balance=balance-100 WHERE id=A AND balance>=100;
-- ↑ check affected rows = 1, 否则 ROLLBACK
UPDATE accounts SET balance=balance+100 WHERE id=B;
INSERT INTO transfers (...) VALUES (...);
COMMIT;

简单、原子、一致。代价是所有账户都要在同一 Postgres 实例。规模上去后做 sharding，跨 shard 必须用方案 B。

方案 B: 没有分布式事务（DynamoDB / 跨 shard）→ Saga + Outbox：

  1. 步骤 1：扣 A 钱（带 idempotency key），同步写一条 transfer_intent 记录（状态=DEBITED）。
  2. 步骤 2：异步 worker 读 intent，给 B 加钱（带同样的 idempotency key），更新状态=COMPLETED。
  3. 步骤 3：失败重试若干次仍失败 → 触发反向补偿：把 A 的钱退回，状态=FAILED。

故障窗口对比：

  • 方案 A：commit 之前任何崩溃都自动 rollback；commit 后 WAL fsync 保证持久化。窗口 ≈ 0。
  • 方案 B：步骤 1 完成、步骤 2 失败这个窗口里，A 看到钱少了但 B 没收到。窗口 = 重试 + 补偿耗时，几秒到几分钟。期间用户会投诉，必须 UI 显示『处理中』。

补偿成本：

  • 方案 A：开发简单，运维成本 = DB 扩容压力。
  • 方案 B：要写 Saga 协调器、idempotency、对账系统（每天扫描悬挂的 intent）、客服流程（处理补偿失败的 case）。一套支付系统 30% 的代码可能在处理失败补偿。这就是 Stripe / 微信支付的工程量来源。

真实选择：金额 + 单区，Postgres 单库事务（甚至 Stripe 早期）；规模 + 跨区，Saga + 外部对账（每天找出不一致并人工干预）。

3. 一张 5 亿行的 Postgres 表，要加一个 NOT NULL 字段。直接 ALTER TABLE 会发生什么？正确做法是什么步骤？

直接 ALTER 会怎样：

  • Postgres 11 之前：ALTER TABLE t ADD COLUMN c TEXT NOT NULL DEFAULT 'x' 会全表重写——每行都要 update 写新版本，5 亿行 × 几 KB = 几 TB 磁盘 IO + 几小时锁表（ACCESS EXCLUSIVE）。生产事故。
  • Postgres 11+：加 column with constant DEFAULT 是 metadata-only（瞬间完成），因为存的是『默认值』，读时返回。但 非常量 default（如 uuid_generate_v4()）仍是全表重写。
  • 不管哪个版本，加 NOT NULL 约束的时候，Postgres 要扫一遍表验证，期间是 ACCESS EXCLUSIVE。

正确步骤（零停机）：

  1. 加列允许 NULL：ALTER TABLE t ADD COLUMN c TEXT，瞬间完成。
  2. 修改应用代码：所有写入都给 c 赋值。
  3. 批量回填：UPDATE t SET c='x' WHERE id BETWEEN ? AND ? AND c IS NULL，每批 1000-10000 行，间隔 sleep，几小时跑完，避免 long transaction 把 vacuum 阻塞。
  4. 加约束 (NOT VALID)：ALTER TABLE t ADD CONSTRAINT c_not_null CHECK (c IS NOT NULL) NOT VALID，瞬间完成，对新写有效但不验证旧数据。
  5. 验证约束：ALTER TABLE t VALIDATE CONSTRAINT c_not_null，只取 SHARE UPDATE EXCLUSIVE 锁（不阻塞读写），扫一遍验证。
  6. 转换为真 NOT NULL（Postgres 12+）：ALTER TABLE t ALTER COLUMN c SET NOT NULL，因为已有 CHECK 约束保证，PG 跳过扫描。

工具化：pg_repack / pg-osc / GitHub gh-ost（MySQL 版）封装了这套流程。

面试加分：能说出『metadata-only』『NOT VALID』『long transaction 阻塞 vacuum』这些细节，立刻拉开档次。

4. 把 DynamoDB 当主存储，听说『按用量付费』很便宜。在什么访问模式下账单会爆炸？至少举 3 种？
  • ① 大批量 scan / 分析查询：DynamoDB Scan 读整个表，1MB 数据 = 大量 RCU。如果你想跑『统计本月新用户』要扫几亿条，账单是 OLTP 主库的 100×。修法：用 streams 导出到 S3 + Athena 跑分析。
  • ② 高基数 GSI：每个 GSI 是独立的索引表，写主表 = 写主表 + 所有 GSI。一个表有 5 个 GSI = 写成本 6×（主表 + 5 索引）。修法：只为真正高频的查询模式建 GSI，能用 sparse index 就 sparse。
  • ③ Hot partition：DynamoDB 按 partition 物理分布，每个 partition 上限 3000 RCU / 1000 WCU。如果有顶流用户的 key 集中在一个 partition，热到了上限就被 throttle。你看到的现象是 ProvisionedThroughputExceededException，被迫上调全表容量（贵）。修法：partition key 加 hash 后缀分散写、用 caching 层挡读。
  • ④ 小 item 但极高 QPS：DynamoDB On-Demand 模式按请求计费，$1.25 / 百万写。一个 10M QPS 的计数器场景，每秒 12.5 美元，每天 100w 美元——这种场景 Redis 远便宜。修法：高 QPS 简单计数器用 Redis 不用 DDB。
  • ⑤ 大 item 频繁更新：DynamoDB 写一个 400KB item 消耗 400 WCU；如果你为了方便把整个用户档案当一个 item 频繁更新，账单会爆。修法：拆字段、避免 partial update 时整 item 写回。

判断原则：DynamoDB 的『便宜』前提是可预测的、点查为主的、小 item 的、固定访问模式。一旦出现 ad-hoc、聚合、热点，账单立刻反超自建。Amazon 自己也大量用 RDS（Postgres/MySQL）而非把所有数据塞 DDB。

5. CAP 定理常被误解为『三选二』。在实际 Postgres / DynamoDB / Cassandra 里，到底牺牲了什么？P 真的是可选的吗？

误解的来源：教科书画三角形，说『选 CP 或 AP，不能要 CA』。其实P（分区容忍）不是选项——网络分区是物理现实，会发生。CAP 真正在问的是：分区发生时，你优先 C（拒绝服务保证一致）还是 A（继续服务但可能不一致）。

具体到各家：

  • 单机 Postgres：CA 看起来满足，但根本不分布式，不在 CAP 讨论范围。一旦上多 replica + 故障切换，就要面对：sync replication（保 C 牺牲 A，主从断连时主库拒写）vs async replication（保 A 牺牲 C，主库继续写但 replica 有 lag）。Postgres 默认 async，偏 A；可配置 synchronous_commit=on + synchronous_standby_names 偏 C。
  • DynamoDB：默认 eventual consistency 读，偏 A。可选 strongly consistent read（多 RCU 2×），分区时该 partition 写可能失败 → 偏 C。事务模式 (TransactWriteItems) 是 CP。
  • Cassandra：可调一致性（tunable consistency）。quorum 读 + quorum 写 = CP-ish（强一致但分区时 minority 节点拒服务）；ONE 读 + ONE 写 = AP（高可用但可能读到旧数据）。开发者按业务场景选每个 query 的 consistency level。
  • Spanner / CockroachDB：偏 CP，靠 Paxos / Raft 共识。分区时 minority 拒写（不能向 minority 提交事务，保证 C），少数派节点不可用（牺牲 A）。Spanner 通过 5 副本 + 跨区域部署，把『不可用』概率压到 5 个 9。

更精细的框架：PACELC（Daniel Abadi 提出）：

  • 分区时：P 选 A 还是 C；
  • 正常时（Else）：选 L（latency）还是 C（consistency）。

例如 DynamoDB = PA/EL（分区时优先可用，正常时优先低延迟，都让步一致性）；Spanner = PC/EC（一直选一致性，付出延迟代价）。这个分类比 CAP 更能区分系统。

面试加分：能区分『分区时选什么』和『正常时选什么』，并举出 PACELC 例子，立刻显得有深度。
