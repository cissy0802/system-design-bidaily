数据库选型 — Postgres、DynamoDB、Cassandra、Redis 怎么选Choosing the Right Database: SQL vs NoSQL, OLTP vs OLAP, ACID vs BASE

问题场景

『选 MySQL 还是 MongoDB』是工程师最容易翻车的决策之一。选错的代价不是性能问题，而是 1-2 年内的迁移地狱：Uber 2016 年从 Postgres 迁回 MySQL（『为什么 Uber 改用 MySQL』那篇博客）、Discord 2017 年从 MongoDB 迁到 Cassandra 再迁到 ScyllaDB、Airbnb 2020 年从 MySQL 拆分出 DynamoDB 承接消息和支付——每次都付出几个工程季度。

本期讲清楚四件事：SQL vs NoSQL 的本质差异、OLTP vs OLAP 的查询模式、ACID vs BASE 的取舍、什么场景选哪个具体产品（Postgres / DynamoDB / Cassandra / Redis）。最重要的认知：『数据库』不是一类产品，是 6-7 类不同范畴的东西。

需求与约束（面试必问）

  • 数据模型：是关系强（订单、支付、社交关系）、还是半结构化（事件流、用户画像）、还是 KV（session、计数器）、还是时序（监控指标）、还是图（推荐、风控）？
  • 读写比与 QPS：1000:1 重读（用 read replica / CDN），1:10 重写（要 LSM-tree 存储引擎），10w QPS 以上单机 RDBMS 撑不住。
  • 数据量与增长率：100GB 单机可以；1TB 进入分片考虑；100TB+ 必须分布式。
  • 一致性要求：账户余额需要 strong consistency；社交动态可以 eventual。
  • 查询模式：点查（GET by PK）、范围扫描、聚合分析、全文搜索、地理查询，每种各有最优引擎。
  • 延迟目标：p99 < 10ms？需要内存型（Redis、DynamoDB）；p99 < 100ms？关系型够用；p99 < 5s？OLAP 可以。
  • 团队熟悉度：能 24×7 oncall 的 DBA 才是真实约束，Cassandra 没人会调你别用。

关键技术点

1. SQL vs NoSQL — 不是『谁更好』而是『建模思路截然不同』

原理：SQL 数据库（Postgres、MySQL、Oracle）以 关系代数 为模型——表 + 主外键，schema-on-write（写时强制结构），靠 JOIN 在查询时拼装。NoSQL（DynamoDB、Cassandra、MongoDB）放弃 JOIN 与多表事务，换取水平扩展、灵活 schema、可预测延迟，本质是『把 join 移到写入时』——通过反范式（denormalization）把要一起读的数据预先放一起。

SQL（Postgres）NoSQL（DynamoDB / Cassandra）Document（MongoDB）
数据模型表 · 行 · 关系KV / 宽列JSON 文档
Schema强制（write-time）无（read-time）灵活
事务多行多表 ACID单行/单分区（有限）4.0+ 多文档事务
水平扩展难（要分片中间件）原生（按 hash key）原生（sharded cluster）
JOIN原生、高效无（要 app 端做）$lookup（慢）
典型 QPS1w-10w 单机百万级（集群）5w 集群
典型场景订单、支付、ERP用户画像、消息、sessionCMS、产品目录
Trade-off：
  • SQL：✅ 关系建模简洁、强一致、JOIN 灵活、生态成熟（ORM/迁移工具/BI）；❌ 单机瓶颈在 1-10w QPS、分片困难、schema 变更要 migration。
  • KV / 宽列（DynamoDB / Cassandra）：✅ 水平扩展无上限、可预测 p99、运维全托管（DDB）；❌ 必须按访问模式建模（access-pattern-first），改一个查询要重建表，不会的人写出来比 SQL 慢 10 倍。
  • Document（MongoDB）：✅ 嵌套对象友好（购物车、用户配置）、schema 灵活；❌ JOIN 弱、聚合慢、事务后引入开销大。
# SQL：关系建模 — 三张表 + JOIN
SELECT u.name, o.total
FROM users u JOIN orders o ON o.user_id=u.id
WHERE o.created_at > NOW() - INTERVAL '7 days';

# DynamoDB：访问模式建模 — 一张表搞定，PK 拼接
# PK=USER#123  SK=ORDER#2026-05-15#ord_abc  -> total, name(冗余)
# Query: PK=USER#123 AND SK begins_with ORDER#
#   一次返回该用户的所有订单, 无 JOIN
ddb.query(
  KeyConditionExpression="PK=:u AND begins_with(SK, :p)",
  ExpressionAttributeValues={":u":"USER#123", ":p":"ORDER#"}
)
现实案例：
  • Uber：早期用 Postgres，规模膨胀后迁回 MySQL（更可控的复制和分片），核心 Schemaless 是 MySQL 上自建的 KV 层。
  • Amazon：购物车、推荐、订单元数据用 DynamoDB；Prime Day 2024 单日峰值 1.46 亿次 DDB 请求/秒。
  • Discord：消息从 MongoDB（4B 文档时挂掉）迁到 Cassandra（180B 消息），再迁到 ScyllaDB（Rust 重写、p99 从 40ms 降到 15ms）。
  • Stripe：核心账本用 MongoDB（早期）+ 自研事务层，后来分片到大量 MongoDB 集群；新业务在评估 TiDB / CockroachDB。
  • Shopify：核心 OLTP 还是 MySQL，按商家分片（pod 架构），单 pod 内是完整 MySQL 集群。

2. OLTP vs OLAP — 一个数据库不可能既快又会分析

原理：OLTP（Online Transaction Processing）面向高并发短事务——一次更新订单 1 行；OLAP（Online Analytical Processing）面向大范围扫描聚合——一次查 1 亿行算月销售额。两者的存储引擎截然不同：

  • OLTP 用行存（row-oriented）：一行数据物理上连续存储，按 PK 取整行 O(1)；典型：Postgres、MySQL、DynamoDB。
  • OLAP 用列存（columnar）：同一列的所有值连续存——扫 1 亿行的 SUM(amount) 时只读 amount 那一列，IO 量是行存的 1/N；同列同类型还能高压缩比（10x+）；典型：ClickHouse、Snowflake、BigQuery、Parquet。
Trade-off：
  • OLTP（行存）：✅ 单行读写极快、事务支持好；❌ 大范围扫描慢（每行都要读完整行）、压缩比低、不适合 BI。
  • OLAP（列存）：✅ 聚合查询快 10-100 倍、高压缩省存储、并行扫描友好；❌ 单行更新昂贵（要重写多个列文件）、点查（按 PK 取一行）慢、事务支持弱。
  • HTAP（Hybrid，混合负载）：TiDB、CockroachDB、SingleStore、Snowflake Unistore 试图『一库两用』，代价是两个存储引擎都要维护、行列同步有延迟。
实战准则：OLTP 跟 OLAP 分开存储。通过 CDC（Debezium / DynamoDB Streams）或定期 ETL 把数据从 Postgres / MySQL 同步到 Snowflake / BigQuery / ClickHouse。不要在 OLTP 上跑 BI 查询——一个慢分析查询能锁死整个生产库。

# 行存（Postgres）扫 1 亿行算总销售：读完整 100M 行 × 50 列 = 50GB IO
SELECT SUM(amount) FROM orders WHERE created_at > '2026-01-01';
-- 全表扫描, 30 秒, 阻塞写入

# 列存（ClickHouse）：只读 amount + created_at 两列, 压缩后约 200MB
SELECT sum(amount) FROM orders WHERE created_at > '2026-01-01';
-- 同样数据量, 200ms, 并行扫描 8 核
现实案例：
  • Netflix：OLTP 用 Cassandra + DynamoDB；分析在 S3 + Iceberg + Trino + Druid（用户实时指标）。
  • Cloudflare：日志 35M events/sec 进 ClickHouse，p99 查询 200ms。开源博客『ClickHouse Capacity Estimation』必读。
  • Uber：OLTP 是 MySQL/Schemaless；分析建在 HDFS + Pinot（实时仪表盘）+ Presto（ad-hoc 查询）。
  • Snowflake：纯 OLAP，存计算分离架构，2024 年存了 6+ EB 客户数据。
  • Shopify Analytics：从生产 MySQL CDC 到 ClickHouse，商家后台报表查询从 30s 降到 300ms。

3. ACID vs BASE — 一致性的两种世界观

原理：

  • ACID（Atomicity、Consistency、Isolation、Durability）是关系型数据库的承诺：事务要么全成要么全失败、并发不互相干扰、提交后绝不丢。实现代价：锁、MVCC、WAL，单机吞吐受限。
  • BASE（Basically Available、Soft state、Eventually consistent）是分布式 NoSQL 的妥协：放弃强事务，换取永远可用 + 水平扩展。典型：DynamoDB（默认 eventually consistent）、Cassandra（可调一致性）、S3（read-after-write 但 list 是 eventual）。

关键：『ACID』和『BASE』不是非黑即白。Postgres 默认 Read Committed 不是 Serializable；DynamoDB 提供 ConsistentRead=true 选项（双倍 RCU）；Spanner / CockroachDB 用 TrueTime / HLC 在分布式系统上做到全局 Serializable。

隔离级别能避免的异常典型数据库默认
Read Uncommitted无（脏读都允许）极少用
Read Committed脏读Postgres、Oracle 默认
Repeatable Read脏读 + 不可重复读MySQL InnoDB 默认
Snapshot Isolation+ Phantom（大多数情况）Postgres、Oracle、SQL Server
Serializable全部Postgres SSI、Spanner、CockroachDB
常见误区：『MySQL 是 ACID 所以一定强一致』错。MySQL 主从异步复制下，刚提交的事务在 replica 上还看不见——这就是 read replica 的 lag。要 read-your-writes 要么读主、要么用 semi-sync replication，要么等 GTID。 
# Postgres 显式事务（ACID）
BEGIN;
  UPDATE accounts SET balance=balance-100 WHERE id=1;
  UPDATE accounts SET balance=balance+100 WHERE id=2;
COMMIT;  -- 两行要么都改要么都不改, 别人看不到中间状态

# DynamoDB 单分区事务（有限 ACID）：TransactWriteItems
ddb.transact_write_items(TransactItems=[
  {"Update":{"Key":{"PK":"ACC#1"}, "UpdateExpression":"SET bal=bal-:v", ...}},
  {"Update":{"Key":{"PK":"ACC#2"}, "UpdateExpression":"SET bal=bal+:v", ...}},
])
# 25 个 item 上限, 2x WCU 成本, 跨表跨区都支持

# Cassandra LWT（Lightweight Transaction）：Paxos 协议, 4x 写代价
UPDATE users SET email='x@y' WHERE id=123 IF email='old@y';
现实案例：
  • Stripe：核心账本必须 ACID，用 MongoDB + 自研 2PC + idempotency key 实现，最近迁向 TiDB 评估。
  • Google Spanner：全球部署的 Serializable RDBMS，靠 TrueTime（GPS+原子钟）做外部一致性，Google 广告系统在用。
  • Amazon DynamoDB：2018 年加入 transactions（最多 25 items），让 Amazon Pay 等业务从 Aurora 迁过来。
  • Airbnb 支付：用 Saga（见 Day 7）+ outbox pattern，在 MySQL 上模拟『跨服务最终一致』。

# 选型伪代码（粗粒度）
def pick_db(req):
    if req.latency_p99 < 1ms or req.is_ephemeral:
        return "Redis"           # session / cache / counter / rate-limit
    if req.query_pattern == "analytics":
        return "ClickHouse / Snowflake"
    if req.needs_joins or req.needs_complex_transactions:
        return "Postgres"        # 90% 的业务从这里开始
    if req.qps > 100_000 and req.access_pattern_is_fixed:
        return "DynamoDB"        # 团队小、要全托管
    if req.data_volume > 100_TB and req.write_heavy:
        return "Cassandra/ScyllaDB"  # 团队大、要 PB
    if req.is_search:
        return "Elasticsearch"
    return "Postgres"            # 默认选它
现实案例：
  • Notion：核心数据全部在 Postgres，2021 年单库到 480 个 logical shards（自研中间件），block 表 20B 行。
  • Figma：Postgres 主库 + 横向分片到 multi-cluster；2023 年博客详述他们如何在不停服下分片。
  • Reddit：Postgres + Cassandra（post + comments）+ Redis（vote count + listing cache）。
  • Lyft：DynamoDB（ride state）+ Postgres（user/profile）+ Redis（geo-index 缓存）+ Druid（分析）。
  • GitHub：MySQL 多个分片（按 repo）+ Redis（counter + queue）+ Elasticsearch（代码搜索）+ ClickHouse（日志分析）。

扩展与优化（增长后怎么办）

  • Postgres 单机 → 分片：先垂直拆（按表）、再水平拆（按 user_id hash）。工具：Citus（已被微软收购）、Vitess（MySQL）、PgCat（连接池+分片）。
  • Read 越来越多：加 read replica；注意 replication lag（见 Day 5），关键读路径走主。
  • Schema 演进：永远 backward-compatible（先加列、再写双份、再读新列、再删旧列），避免长事务的 ALTER TABLE。工具：gh-ost、pt-online-schema-change。
  • 引入第二种 DB：典型路径 = Postgres + Redis（缓存 + queue）→ 加 ClickHouse（分析）→ 加 Elasticsearch（搜索）。
  • 多区域：DynamoDB Global Tables、Spanner、CockroachDB 原生支持；Postgres 跨区要么 logical replication（异步）要么 BDR（贵）。
  • 瓶颈识别：监控连接数、慢查询（pg_stat_statements）、buffer cache hit ratio、replication lag、磁盘 IOPS。

常见陷阱（资深工程师也常翻车）

1. 把 NoSQL 当 SQL 用：在 DynamoDB 上做 scan + filter（不走 PK），单次查询消耗几千 RCU、延迟几秒——这是用法错了，不是 DDB 慢。NoSQL 必须『访问模式优先』建模。
2. MongoDB 当真相源跑账户：4.0 之前没有跨文档事务，多个集合更新可能错乱。即使有事务，分片集群下成本也高，账务系统不建议。
3. Redis 当 source of truth：RDB / AOF 持久化是 best-effort，宕机会丢秒级数据。账户余额要写穿到 Postgres，Redis 只做读路径加速。
4. 在 OLTP 上跑大分析：一个 SELECT COUNT(*) FROM orders 全表扫，能把生产 RDS 锁死、replication 卡死、p99 跳到几秒。务必走 read replica 或异步 ETL 到仓库。
5. 跨表事务 + 微服务：把两个 service 的数据库放进一个分布式事务是反模式。用 Saga / Outbox（见 Day 7）做最终一致。
6. 过早选 Cassandra / Kafka / Elasticsearch：『大厂都用所以我用』是糟糕的理由。运维一个 Cassandra 集群至少要 2 个全职 SRE，10 人小团队应该用 Postgres + 托管缓存撑到 1M MAU 再考虑。

English Summary

Database selection is a long-term decision: migrations cost engineering quarters. SQL (Postgres/MySQL) gives you relational modeling, ACID transactions, and JOINs, capped around 10-50TB and 100k QPS single-node. NoSQL (DynamoDB/Cassandra) trades joins and multi-row transactions for horizontal scale, predictable p99 latency, and access-pattern-first modeling. OLTP and OLAP are fundamentally different storage engines (row vs columnar) and should be separated — never run BI on the production transactional DB. Redis is a cache, not a source of truth. Default to Postgres and only adopt specialized stores (Cassandra, DynamoDB, ClickHouse, Elasticsearch) when measurable bottlenecks demand them.

深入思考（点击展开答案）

1. 为什么列存的 OLAP 系统聚合查询能比行存快 10-100 倍？这个『10-100x』里有几个独立来源？

不是『一个原因 100 倍』，而是『几个 2-3 倍叠加』。

  • IO 量减少（5-50x）：50 列的表只读 2 列做 SUM(amount)，理论上 IO 缩 25 倍；实践中由于 page 对齐和压缩，3-10 倍很常见。
  • 压缩率（3-10x）：同一列同类型数据熵低，dictionary encoding、RLE、bit-packing 压到 1/10。读取磁盘 IO 又少一个量级。
  • 向量化执行（2-4x）：列存可以让 CPU 用 SIMD 一次处理 8-16 个值（AVX-512），相比逐行处理快几倍。ClickHouse、DuckDB 都是 vectorized engine。
  • 谓词下推 + 跳过 block（2-10x）：列存按 chunk（如 8K 行）存储每列 min/max，WHERE date > '2026-01-01' 直接跳过不相关 chunk，IO 又减少。
  • 并行扫描（4-16x）：列存天然适合多 CPU 并行（每个 CPU 扫一段），行存因 row lock / shared structure 并行难。

叠起来轻松 100 倍。但反过来——列存做 PK 点查（取一整行 50 列）要从 50 个列文件各读一段再拼装，反而比行存慢 10 倍。所以两类引擎不是『谁取代谁』，是各干各的。

2. 在 DynamoDB 上你刚学会 single-table design 把所有实体放一张表。3 个月后产品要『按 email 查 user』，原表 PK=USER#<uid>。你有几种解法，各自代价？
  • ① GSI（Global Secondary Index）：建一个 GSI，PK=EMAIL#<email>。代价：GSI 异步复制有秒级 lag、写入要双倍 WCU（主表 + GSI）、GSI 不支持 strong consistent read。最常见的解法。
  • ② Scan + Filter：直接全表扫。代价：千万行 = 几十秒 + 几千 RCU 成本，每次 $$$，绝对不能上生产。
  • ③ 反向 item：在原表插一行 PK=EMAIL#<email> SK=USER, uid=...，应用层维护 user 和 email 项的双写。代价：自己维护一致性（双写有可能不同步）、占主表 storage。早期 DDB 没 GSI 时常用，现在不推荐。
  • ④ 外置二级索引：写 ElasticSearch 或 Redis 维护 email → uid 映射。代价：多一个系统、CDC 链路要可靠（DDB Streams → Lambda → ES）、最终一致。
  • ⑤ 重新设计 PK：极端情况下改用 email 当 PK，uid 走 GSI。代价：完全推翻历史，所有访问改路径，几乎不可行。

实战：99% 的情况是 ①。但要在设计阶段就预判未来的访问模式，避免事后改造。这就是『access pattern first』的含义——不是省事，是 DDB 模型逼你提前想清楚。

3. 『Postgres 单机能撑到多少 QPS / 多大数据量』这个问题，正确的回答方式是什么？10w QPS 是哪个维度的瓶颈先到？

错误回答：『10w QPS』『1TB』——任何固定数字都是错的，因为瓶颈维度取决于负载特征。正确回答是：

  • 『取决于读写比、行大小、查询复杂度、磁盘类型』。
  • 用 r6id.32xlarge（128 vCPU + 1TB RAM + NVMe）跑纯点查能到 30w+ QPS；同机器跑 5-table JOIN + GROUP BY 可能只有 2k QPS。差 150 倍。

10w QPS 谁先撑不住：

  1. 连接数：Postgres 每连接一个 OS 进程 + 10MB 内存，1000 连接就累。先撞这个——靠 PgBouncer 池化解决。
  2. WAL（写日志）写入：纯写场景下 WAL flush 是瓶颈，10w writes/sec 接近 NVMe IOPS 上限。靠 batch commit + async commit 缓解。
  3. 锁竞争：热行（单个用户、单个产品）的更新会因 row lock 排队，QPS 跟单 row 的并发更新无关。靠 sharding 单热行（counters 拆 16 个）。
  4. Vacuum / autovacuum：MVCC dead tuple 累积，autovacuum 一启动就抢 IO，p99 抖动严重。
  5. Replication lag：异步复制下，主库写太快 replica 跟不上，read replica 变 stale，影响读路径正确性。

面试时这样回答能立刻把对方拉到『真实工程』层面，胜过任何数字。

4. 同样是『最终一致』，DynamoDB 默认 read 和 S3 早期的 list-after-write 都属于。但实际工程里两者表现差异巨大，为什么？

关键差异在于：『最终』有多远，以及读到旧数据时业务能否容忍。

  • DynamoDB eventually consistent read：异步复制到 3 个 AZ 的延迟通常 < 1 秒，绝大多数请求看到的是最新值。少数情况下读到几百 ms 前的旧值。业务上一般可接受（feed 慢 1s 没事），且 DDB 提供 ConsistentRead=true 给关键路径用（双倍成本）。
  • S3 list-after-write（2020 年之前）：put 一个新对象后立刻 list 可能看不见，且窗口长达几十秒。这种 negative case（『刚写的找不到』）比 stale read 难处理——业务往往会重试，导致重复写入；批处理依赖 list 来枚举工作单元就会丢任务。2020 年 12 月 AWS 把 S3 升级到 strong read-after-write + strong list consistency，就是因为这种 gap 实在太难用。

更普遍的判断：

  • 『缺值』比『旧值』难处理（读到 null 业务会去插入，造成重复）；
  • 读后写依赖（read-modify-write）会因 stale 读导致 lost update；
  • 多 key 查询的最终一致更可怕——你可能看到 A=v2 但 B 还是 v1，违反不变量。

面试加分：能区分 monotonic read、read-your-writes、causal 等具体一致性模型（见 Day 6），而不是只说『eventual』。

5. 一个产品现在用 Postgres，全表 5 亿行，BI 每天跑全表聚合，把生产库 p99 拖到 3 秒。你被叫去『救火』，列出 4 个量级不同的解法，从『5 分钟』到『3 个月』。
  • ① 5 分钟级（止血）：把 BI 用户的 statement_timeout 设到 30s、把 BI 跑的 SQL 放到一个独立的 PG role，限制 max_connections=5。立刻不让 BI 把整个库拖死。代价：BI 报表跑不完，但生产 OLTP 救回来了。
  • ② 1 天级（架构隔离）：BI 查询全部切到 read replica（physical replication）。生产 OLTP 完全不受影响。代价：replication lag（几秒-几分钟）下 BI 看到的不是最新数据；replica 自己有时被分析查询打挂，但不影响主库。这是大多数中型公司的常态方案。
  • ③ 1 周级（CDC 到 OLAP）：用 Debezium 监听 Postgres logical replication，把数据流到 ClickHouse（或 BigQuery / Snowflake）。BI 完全脱离 PG。代价：多一套系统要维护、schema 变更要同步、最终一致（分钟级延迟）。但 BI 查询从 30s 降到 200ms。
  • ④ 1-3 个月级（数据仓库 + Lakehouse）：搭建完整数仓 / lakehouse（Iceberg + Trino / Databricks），CDC + 定时 batch ETL 双轨；BI / 数据科学 / ML feature 全在数仓里跑；生产 PG 退守为纯 OLTP。这是大型公司的终态。代价：需要数据工程团队、$$$、metadata 治理是另一个大坑。

关键启示：『一个数据库解决所有问题』是初创期的便宜假设，规模上来必然分裂为 OLTP + OLAP + cache + search 等多套。面试中能讲清这条演进路径，比讨论某一种 DB 本身更体现 staff+ 水平。
