数据库选型 — 为什么不能『默认上 Postgres』也不能『默认上 MongoDB』Database Selection: SQL vs NoSQL, OLTP vs OLAP, ACID vs BASE, Choosing Among Postgres / DynamoDB / Cassandra / Redis

问题场景

面试官说『设计一个 Instagram』，你脱口而出『用 MongoDB』——这是错答案的代名词。数据库选型不是品牌偏好，是对访问模式、一致性需求、规模和成本的精确建模。Uber 2016 年从 Postgres 迁到自研 Schemaless（底层 MySQL）省了一大笔；Discord 2017 年从 MongoDB 迁到 Cassandra，因为 1.2 亿消息/天把 MongoDB 的 working set 推爆；Stack Overflow 至今所有数据放在 1 个 SQL Server 实例上跑得飞快。

本期讲四件事：关系 vs 非关系、OLTP vs OLAP、ACID vs BASE、四种代表性 DB 的『何时选』。重点不是背特性，是建立一个选型决策框架。

需求与约束（面试必问）

  • 数据模型：实体关系强（用户/订单/商品）还是文档型（嵌套结构、schema 经常变）？图结构（社交关系/反欺诈）？时序（监控指标）？
  • 读写比与 QPS：读 100 万 QPS / 写 1 万 QPS（feed）vs 读写 1:1（即时通讯）。
  • 数据量：100GB（单机 Postgres 轻松）vs 100TB（必须分片/列存）vs 10PB（数据湖）。
  • 一致性要求：转账必须强一致；点赞计数容忍秒级 stale。
  • 延迟 SLO：交易类 p99 < 50ms；分析类查询 30s 可接受。
  • 查询模式：点查（按 key）、范围扫描、聚合、全文搜索、join？
  • 跨地域：单区域？多区域读写？强一致需求决定是否能上 Spanner / CockroachDB。
  • 团队熟悉度 + 运维成本：能不能招到会调优的人？Cassandra 的运维不是 Postgres 能比的。

关键技术点

1. SQL vs NoSQL：本质是『预定义 schema + join + 事务』vs『水平扩展 + 灵活 schema』

原理：SQL（关系型）建立在关系代数与 ACID 之上，核心能力是 join、事务、约束。NoSQL 不是一类，是一个反义词集合——KV（DynamoDB/Redis）、文档（MongoDB）、宽列（Cassandra/HBase）、图（Neo4j）。共同特点是放弃部分 SQL 能力换取扩展性或灵活性。

维度SQL（Postgres/MySQL）NoSQL（DynamoDB/Cassandra/Mongo）
Schema强 schema，DDL 迁移schema-less 或动态 schema
Join强项多数不支持，靠 app 端拼
事务多行 ACID多数单行/单分区原子；DynamoDB 有限事务
水平扩展难（需手动分片）原生（hash partition）
查询灵活性任意 SQL按主键 + 二级索引，临时聚合差
一致性默认强一致多数最终一致，可配
Trade-off：
  • SQL：✅ 灵活查询、强一致、生态成熟（ORM/BI/运维工具齐全）；❌ 单机上限（即便 Postgres 也就 64 核 + 几 TB 数据）、跨表事务在分片后崩塌。
  • NoSQL：✅ 水平扩展线性、写吞吐高、schema 演进自由；❌ 查询模式必须提前设计（建错了 access pattern 几乎无救）、跨实体一致性靠 app。
  • 『新型 SQL』（NewSQL/Distributed SQL）：CockroachDB / Spanner / TiDB / YugabyteDB——SQL + 水平扩展 + ACID，代价是写延迟高（Raft 多副本 commit ~10-50ms）、运维复杂、成本高。

ORDER BY created_at DESC LIMIT 50;

现实案例：
  • Stack Overflow：所有问答数据 ~400GB 放在单实例 SQL Server，配高内存 + SSD，全球 1.3 亿 PV/月轻松扛住。证明『SQL 单机』比想象的强。
  • Shopify：核心订单/商品用 MySQL（pod 化分片，每个商家一个 pod），靠应用层 routing。
  • Airbnb：早期单 MySQL，2014 年开始按业务垂直拆库（users / listings / reservations）。
  • Discord：消息从 MongoDB 迁到 Cassandra（2017）再迁到 ScyllaDB（2022），因为 MongoDB 的 B-tree + working set 模型撑不住 1.2 亿/天的写。
  • Instagram：核心元数据 Postgres + 分片（user_id hash）；feed 用 Cassandra；session 用 Memcached。

2. OLTP vs OLAP：行存还是列存，不仅是『谁更快』

原理：OLTP（Online Transaction Processing）服务用户实时操作：单条记录读写、事务、低延迟（<100ms）、高并发（千-万 QPS）。OLAP（Online Analytical Processing）服务报表/BI/数据科学：跨百万-亿行聚合、扫描密集、复杂 join、延迟容忍秒到分钟。

关键差异是存储布局：

  • 行存（Postgres/MySQL/Oracle）：一行的所有列连续存储。优势：点查/单行更新极快（一次磁盘 seek 拿到全部字段）。劣势：SELECT SUM(amount) FROM orders 要读全部行，浪费 IO。
  • 列存（ClickHouse/Snowflake/BigQuery/DuckDB/Parquet）：每一列单独存储。优势：聚合只读相关列，压缩率极高（同列同类型，压缩比 10-100x），向量化 CPU 处理（SIMD）。劣势：单行写入慢（要 touch N 个文件）、点查弱。
OLTP (行存)OLAP (列存)
典型查询WHERE id=X，UPDATE … WHERE id=XGROUP BY country, SUM(revenue)
表大小GB-TBTB-PB
并发千-万 QPS几十-几百
事务必需大多无（追加为主）
压缩比2-3x10-100x
代表Postgres, MySQL, OracleSnowflake, BigQuery, ClickHouse, Redshift
Trade-off： 不要在 OLTP 上跑 OLAP 查询——一个 SELECT SUM 全表扫描会把生产 DB 的 buffer pool 冲掉、IO 打满、p99 飙升。标准做法：CDC（Debezium / DMS / 内置 logical replication）把 OLTP 数据持续同步到 OLAP 仓库；BI 工具只查仓库。延迟从分钟到秒不等。
现实案例：
  • Netflix：在线播放元数据 Cassandra；用户行为流入 Kafka → S3 → Snowflake/Iceberg 做 OLAP 分析。
  • Uber：行程数据 OLTP 在 Schemaless（MySQL 之上），分析在 Pinot（实时 OLAP，sub-second 查询数十亿行）+ Hive。
  • Cloudflare：边缘日志（每秒 5000 万事件）写 ClickHouse，单集群查询百亿行只要秒级。
  • Pinterest：用户行为 → Kafka → Druid（实时 OLAP），用于 dashboard 和实验分析。

3. ACID vs BASE：不是非此即彼，是『按数据重要性配置』

原理：

  • ACID（Atomicity, Consistency, Isolation, Durability）：传统 RDBMS 的保证。事务要么全成功要么全失败，commit 后永不丢失，并发事务看起来像串行。
  • BASE（Basically Available, Soft state, Eventually consistent）：分布式系统因网络分区不可避免选择牺牲一致性换可用性（CAP 中 AP 系统）。

关键认知：同一个系统里不同数据可以选不同保证。账户余额必须 ACID，点赞计数可以 BASE。盲目要求全局 ACID 会让你付出 10 倍延迟成本。

Repeatable Read+ 不可重复读幻读（标准里）MySQL InnoDB

Trade-off：
  • 强一致代价：跨节点强一致要 Paxos/Raft 一轮（典型 10-50ms），跨区域上百 ms；Spanner 用 TrueTime 也要 commit wait ~10ms。
  • 最终一致代价：业务复杂度——必须设计幂等、补偿、对账。Stripe 内部对每笔支付有 7 道对账流程。
  • 误用 ACID：把『更新用户最后登录时间』也放进事务里，热行竞争把整个事务拖到秒级。
现实案例：
  • Stripe：账本（ledger）严格 ACID（Postgres，单写主库），辅助数据（webhook 状态、metrics）BASE。
  • Amazon：购物车曾用 Dynamo（BASE），允许『添加冲突合并』（两次添加同一商品都保留），换取永远可用。
  • Google Spanner：AdWords 全球账单系统跨大洲强一致，靠原子钟 + GPS 的 TrueTime 把不确定性压到 7ms。
  • Twitter：tweet 写入是『最终一致』fanout，但用户登录 + 关注关系是 ACID（Manhattan 的 strong 模式）。

4. 四种代表性引擎：何时选 Postgres / DynamoDB / Cassandra / Redis

选型一句话规则：
  • 有复杂查询、join、事务、数据量 < 10TB → Postgres。能用 SQL 解决就别引入新 DB。
  • 访问模式固定（用 PK 查）、要全球扩展、按需付费 → DynamoDB。但 access pattern 必须提前设计死。
  • 写密集、时序、有时间维度的『按 user 追加』场景（消息/日志/事件） → Cassandra / ScyllaDB。
  • 需要 < 1ms 延迟、数据结构（list/set/sorted set/stream） → Redis。但当 cache，不当 source of truth。
现实案例：
  • Postgres：Instagram 元数据、Notion 文档树（递归 CTE 查层级）、GitLab 全站、HEY 邮件、Linear。
  • DynamoDB：Amazon 购物车（始祖）、Lyft 司机位置、Disney+ 用户配置、Nike 会员、Capital One 信用卡交易。
  • Cassandra：Netflix 观看历史（每秒数百万写）、Apple iMessage 元数据（百 PB 级）、Instagram 早期 feed、Uber 行程数据。
  • Redis：Twitter timeline cache、Stripe rate limiter、GitHub Actions 队列、Discord presence、Pinterest 推荐 feature store。

扩展与优化（增长后怎么办）

  • Postgres 撑不住 → 先垂直拆库（按 bounded context：users / orders / billing），再 sharding（pg_partman、Citus、Vitess）。
  • 引入读副本：90% 业务读写比 9:1，读副本能立刻把主库压力降 80%；但要处理 replication lag（read-your-writes 问题）。
  • OLTP + OLAP 分离：CDC（Debezium）流出到列存仓库；BI 永远不要打 OLTP。
  • 引入专用 DB：搜索 → Elasticsearch / OpenSearch；图 → Neo4j / DGraph / TigerGraph；时序 → InfluxDB / TimescaleDB / Prometheus；向量 → Pinecone / Weaviate / pgvector。
  • 多区域：单写主 + 跨区 replica；或上 Spanner / Cockroach / Aurora Global。代价：写延迟从 1ms → 100ms+。
  • 瓶颈识别：监控 IOPS、CPU、connection pool、slow query、locks。Postgres 看 pg_stat_activity、pg_stat_statements。

常见陷阱（资深工程师也常翻车）

1. 用 MongoDB 当默认：『schema 灵活』听起来好，几年后你会发现没有 schema 等于每个字段都有 5 种可能的格式。Foursquare 2010 年因为 MongoDB 没 schema 验证导致脏数据酿成 11 小时宕机。
2. DynamoDB 不预先设计 access pattern：『先建表，以后再想怎么查』是死路。DynamoDB 没有自由 query，PK + SK + GSI 都得在写入前算好。Rick Houlihan 的 talk 反复强调：DynamoDB 是『反范式 + 单表设计』。
3. Cassandra 当 OLTP 关系库用：Cassandra 没有 join、二级索引很差、count(*) 不支持。它是『为已知 query 优化的 KV+宽列』，不是 RDBMS 替代品。
4. Redis 当 source of truth：RDB / AOF 持久化是『最多丢 1 秒』而不是 0 丢失。账户余额、订单状态绝不能仅放 Redis。
5. 跑 SELECT * FROM big_table 在生产：全表扫描把 buffer pool 冲掉、connection 占住、replication lag 暴涨。OLAP 查询永远走仓库。
6. UUID 作主键：随机 UUID 让 B-tree 索引页面随机插入，写放大严重、缓存命中率差。改用 UUIDv7 / ULID / Snowflake（时间有序）。

English Summary

Database selection is not a brand preference — it's a precise mapping from access patterns, consistency requirements, scale, and cost to the right storage engine. Use Postgres by default for relational, transactional workloads under 10TB; DynamoDB when access patterns are fixed and you need elastic scale; Cassandra for write-heavy, time-ordered, append-mostly data; Redis as a cache or for low-latency data structures, never as source of truth. Separate OLTP from OLAP via CDC — never run analytical queries on your transactional DB. ACID vs BASE is not all-or-nothing: pick the right guarantee for each piece of data.

深入思考（点击展开答案）

1. 一个团队最常见的『默认选 MongoDB』决策为什么大概率会后悔？至少列 4 个二阶问题。

『schema-less』的吸引力是短期的，长期成本会从四个维度反噬：

  • 没 schema 等于 schema 散落在 100 处代码：每个 reader 都要做 obj.field || obj.field_v2 || obj.legacy_field。Foursquare 2010 年 11 小时宕机就是因为没有 schema 验证让 shard 大小估计错误。
  • JOIN 缺失逼着反范式：把 user 信息嵌入到每条订单文档里——user 改名要更新百万订单。或者 app 端做 $lookup，性能和复杂度都被推到应用层。
  • 事务弱：MongoDB 4.0 才支持多文档事务，且性能很差（10x slower）。需要事务的业务（账单）实际不能用。
  • 查询规划器不可控：复杂 query 的 explain plan 比 Postgres 难读；hint 机制不完善；新版本可能性能退化（5.0 → 6.0 多次回归案例）。
  • 分析能力差：无 SQL = BI 工具体验糟糕；最终还是要 CDC 到仓库。
  • License 风险：MongoDB 2018 切到 SSPL，主流云厂只能提供旧版本 fork（DocumentDB），社区分裂。

更深的教训：『schema-less 灵活』≠『架构灵活』。真正的灵活来自清晰的领域建模 + 演进式迁移工具，不是把锅甩给 reader。

2. Spanner / CockroachDB 这种『SQL + 全球强一致』听起来无敌，为什么没有取代所有 OLTP？

『又要 SQL 又要 ACID 又要水平扩展』的代价比表面大：

  • 写延迟天花板：跨副本 Raft commit 至少一个 RTT；跨区域强一致是 100ms+。Postgres 单机 1ms 的写延迟，CockroachDB 跨区是它的 100 倍。延迟敏感业务（购物车 add-to-cart）直接体验劣化。
  • 事务模型不同：Spanner / Cockroach 用 MVCC + Optimistic concurrency，热行竞争表现极差（contention abort），不是『Postgres 拿过来横向扩展』那么简单。Stripe 在 2022 年公开抱怨 CockroachDB 在 hot account 上的尾延迟。
  • 成本：Spanner 是 Cloud SQL 的 5-10 倍价格；CockroachDB 自托管运维门槛高。
  • 生态：Postgres 30 年的 extension（PostGIS、pgvector、TimescaleDB）、ORM、监控工具，Spanner/Cockroach 远不能比。
  • 真正『全球』需求很少：大多数业务可以接受『单区域写主 + 异步跨区 replica』，强一致需求往往局限在账本类小子集——这部分可以单独跑 Spanner，其他用 Postgres。

结论：NewSQL 是特定子集问题的好答案（全球账本、跨大洲库存），不是 Postgres 替代品。

3. DynamoDB hot partition 在『按用户 id 分区』时会出现。你的产品有一个用户占总流量 5%，怎么修？为什么 GSI 不解决问题？

背景：DynamoDB 每 partition 上限 3000 RCU / 1000 WCU。大 V 用户写 1500 WCU 就直接 throttle。

GSI 为什么不解决：GSI 是另一份异步复制的数据，仍然按其 PK 分区——你把 GSI 的 PK 设成同样的 user_id，热点依然在同一个 GSI partition。GSI 解决的是『不同 access pattern』，不是『负载分散』。

真正的修法：

  • Write sharding：把 PK 从 USER#123 改成 USER#123#shard{0..31}，写时随机选 shard。读时要做 32 次 query 合并（读放大 32 倍）。
  • 异步聚合：单点写入 Kinesis / DynamoDB Stream，下游 Lambda 聚合后再写最终 partition，把瞬时尖峰平滑掉。
  • 分离热数据：top 1% 大 V 用户走单独的『hot tier』表（甚至换引擎，比如 Aurora 或 ScyllaDB），不和长尾用户共用容量。Twitter 长期这么做（celebrity-only 路径）。
  • Adaptive capacity：DynamoDB 2018 引入了自适应容量，会把空闲 partition 的 RCU/WCU 借给热 partition，但是有上限（一个 partition 物理硬上限不变），只能缓解不能根治。
  • Pre-compute：很多『大 V 流量』其实是读公开数据——上 CDN / 边缘 cache，DynamoDB 根本不该承担。
4. 同样是『日志/事件存储』，Kafka、Cassandra、ClickHouse、S3 + Parquet 都能做。各自的边界在哪？

这四个不是替代关系，是数据生命周期的不同阶段：

  • Kafka：『流动中的数据』。append-only log，保留 1-7 天，主要是分发与解耦（producer → consumer），不是查询引擎。LinkedIn 用 Kafka 每天处理 7 万亿条消息，但你不会去 Kafka 里 SELECT。
  • Cassandra / ScyllaDB：『近期可查的事件』。LSM-tree，写入 1M+ TPS，按 partition key + clustering key 范围扫描。Discord 把过去 1-90 天消息放 Cassandra，能按 channel + time 范围查。但聚合差，count(*) 不可用。
  • ClickHouse / Druid / Pinot：『近实时分析』。列存 + MergeTree，写入 1M TPS，扫描数十亿行做 GROUP BY 聚合在秒内。Cloudflare 把边缘日志（每秒 5000 万）写 ClickHouse，用于 dashboard 和 ad-hoc 分析。
  • S3 + Parquet + Iceberg / Delta：『冷数据 + 历史归档』。存储便宜（$0.023/GB-mo），Snowflake / Athena / Trino 查询，延迟分钟到小时，但 PB 级成本最低。Netflix 总数据 100PB+ 大部分在这层。

典型流水线：app → Kafka（缓冲）→ Cassandra（在线查询）+ ClickHouse（实时分析）→ S3 Iceberg（长期归档 + ML 训练）。每层覆盖不同 latency/cost 维度。

5. 你接手一个『一切都放在 MySQL 单库 800GB』的系统，QPS 已经接近上限。给出一个 12 个月的演进路线，按风险与收益排序。

不要一上来就讲分片。按风险从低到高、ROI 从高到低排序：

  1. 0-1 月：profiling + 慢查询治理（最高 ROI）。打开 slow_query_log，pt-query-digest 找 top 10 慢 query；加索引、改 SQL（避免 SELECT *、N+1、不必要的 OFFSET 大分页）。多数系统这一步就能腾出 50% 容量。风险：极低。
  2. 1-2 月：引入 Redis cache。热点读（用户资料、商品详情、配置）走 cache-aside，命中率到 95%+ 后 DB 读 QPS 大幅下降。风险：低，需要处理失效一致性。
  3. 2-4 月：读写分离 + 读副本。读副本承接报表、批处理、慢查询；写仍主库。注意 read-your-writes（发帖后立即读自己的帖子要走主）。风险：中，需要业务区分『关心 lag 的读』。
  4. 4-6 月：CDC + 数据仓库分流 OLAP。Debezium 把 binlog 同步到 Snowflake / ClickHouse，BI、报表、数据科学全部从仓库走，OLTP 不再被分析查询冲击。风险：中，需要数据团队适配。
  5. 6-9 月：垂直拆库（bounded context 拆）。按业务把 users / orders / billing / cms 拆到独立 MySQL 实例。跨库 join 改 app 端拼接或服务调用。这一步要求服务化先行，风险：高，需要明确 owner。
  6. 9-12 月：水平分片（Vitess / Citus / 自研）。最大的几张表（订单、交易流水）按 user_id hash 分片。这是真正的『大手术』，事务模型、跨片 query、ID 生成、迁移工具全要做。风险：极高，能不做就不做。

关键判断：很多团队跳过前 4 步直接搞分片，结果折腾 1 年发现『当初加几个索引就够了』。Stack Overflow 至今未分片，因为他们花了 10 年把前 4 步做到极致。
